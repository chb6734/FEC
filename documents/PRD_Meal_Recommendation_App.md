## 1. 문서 개요

### 1.1 제품 개요

| 항목 | 내용 |
|------|------|
| **제품명** | 식사 추천 앱 (가칭) |
| **플랫폼** | iOS (Swift/SwiftUI) |
| **한 줄 정의** | 식사시간에 맞춰, 사용자의 과거 패턴을 학습하여 개인화된 음식을 추천하는 앱 |
| **MVP 목표** | "무엇을 먹을지 고민하는 사용자"에게 즉시 추천을 제공하고, 사용 이력 기반 개인화 품질을 검증 |

### 1.2 타겟 사용자

| 페르소나 | 설명 | 핵심 니즈 |
|----------|------|-----------|
| **식사 결정 고민족** | 점심/저녁 메뉴 선택에 시간을 쓰는 직장인·학생 | 빠른 추천, 반복 메뉴 회피 |
| **다이어트/식단 관리자** | 제한사항(칼로리, 알레르기)이 있는 사용자 | 맞춤 필터, 제한사항 반영 |
| **새로운 맛 탐색자** | 비슷한 음식만 먹는 패턴에서 벗어나고 싶은 사용자 | 다양한 추천, 선호도 학습 |

### 1.3 제품 목표 (목표-단계-지표)

| 목표 | 단계 | 지표 |
|------|------|------|
| **가설 검증** | 4주 베타 | D1 리텐션 30%+, 추천 클릭률 20%+ |
| **개인화 품질** | 4주 베타 | 로깅 완료율 50%+, 선호도 피드백 활용률 30%+ |
| **PMF 탐색** | Phase 2 | MAU 1,000+, WAU/MAU 40%+ |

## 2. 문제 정의 & 사용자 가치

### 2.1 해결할 문제

1. **식사 메뉴 선택 부담:** "뭐 먹지?" 고민에 소요되는 시간·스트레스
2. **반복적 선택:** 익숙한 메뉴만 반복, 다양성 부족
3. **제한사항 관리:** 알레르기·다이어트 등이 반영되지 않은 추천의 한계

### 2.2 사용자 가치 제안

> **"식사시간에 앱을 열면, 나에게 맞는 음식이 즉시 제안된다. 선택할수록 더 맞는 추천이 된다."**

| 가치 | 구체적 설명 |
|------|-------------|
| **즉시성** | 온보딩 후 첫 화면부터 추천 제공 |
| **개인화** | 취향·제한사항·시간대 반영 |
| **학습** | 선택 이력 축적으로 추천 품질 개선 |

## 3. 기능 요구사항

### 3.1 기능 우선순위 매트릭스

| 우선순위 | 기능 | 검증 가설 |
|----------|------|-----------|
| **P0** | 온보딩 | 3~5문항으로 Cold Start 해소 가능 |
| **P0** | 식사시간별 추천 | 시간대 기반 추천이 사용자 의도와 일치 |
| **P0** | 식사 로깅 | 로깅 완료율로 학습 데이터 확보 가능성 검증 |
| **P0** | 로컬 DB + 규칙 기반 엔진 | DB 저장(하드코딩 금지), 50~100개 음식, 이미지 DB 저장 |
| **P1** | 선호도 피드백 | 피드백이 추천 품질 개선에 기여 |
| **P1** | 설정 (식사시간·제한사항) | 커스텀 필요성이 충분한지 검증 |

### 3.2 P0 기능 상세

#### F1. 온보딩

| 항목 | 내용 |
|------|------|
| **목적** | Cold Start 해소, 초기 프로파일 수집 |
| **플로우** | 앱 최초 실행 → 3~5개 질문 → 메인 화면 진입 |
| **필수 문항 (3개)** | ① 식사 패턴 (아침/점심/저녁 중 주로 먹는 시간대) ② 식이 제한사항 (없음/채식/글루텐프리/저칼로리/알레르기 등) ③ 선호 요리군 (한식/중식/일식/양식/분식/기타) |
| **선택 문항 (2개)** | ④ 평소 식사 비용대 ⑤ 싫어하는 음식/재료 (자유 입력) |
| **완료 조건** | 필수 3문항 응답 완료 시 다음 단계 가능 |
| **저장** | UserDefaults 또는 CoreData에 프로파일 저장 |

#### F2. 식사시간별 추천

| 항목 | 내용 |
|------|------|
| **목적** | 현재 식사시간대에 맞는 음식 **3개** 추천 |
| **식사시간 정의** | 아침 6–10시, 점심 11–14시, 저녁 17–21시 (설정에서 1개 커스텀 가능) |
| **추천 로직** | ① 현재 시간대 매칭 ② 프로파일(제한사항·선호) 필터링 ③ 최근 선택 이력 회피 ④ 점수 기반 정렬 |
| **노출 형식** | **카드형 스택**, 틴더 스타일 좌/우 스와이프. 카드 구성: 중앙 음식 사진/그래픽 + 색상 테두리 |
| **동작** | 스와이프로 카드 넘기기, 추천 카드 탭/선택 시 → 식사 로깅 플로우 또는 상세 보기 |

#### F3. 식사 로깅

| 항목 | 내용 |
|------|------|
| **목적** | 사용자 선택 이력 수집 → 추천 품질 개선 |
| **진입 경로** | ① 추천 카드에서 선택 ② "직접 입력" 버튼 |
| **입력 항목** | 음식명, 식사시간대, (선택) 비고 |
| **저장** | 로컬 DB에 이력 저장 |
| **결과** | 로깅 완료 시 메인(추천) 화면으로 복귀 또는 축하 메시지 |

#### F4. 로컬 DB & 추천 엔진

| 항목 | 내용 |
|------|------|
| **DB 규모** | MVP: 50~100개 음식 |
| **저장 방식** | **DB 저장 필수** (CoreData 등). 하드코딩(JSON 번들, plist, 코드 상 상수) 금지 |
| **음식 스키마** | id, name, category, mealType[], restrictions[], tags[], baseScore, **imagePath** (nullable, 추후 이미지 저장) |
| **추천 알고리즘** | 규칙 기반 점수화: `시간대 가중치 + 취향 매칭 + 최근 선택 회피` |
| **이미지** | 운영자가 직접 DB에 저장. MVP 시점부터 imagePath 필드 포함, 추후 이미지 파일도 DB 저장 |

### 3.3 P1 기능 상세

#### F5. 선호도 피드백

## 5. 사용자 플로우

### 5.1 첫 사용자 (Cold Start)

```
앱 실행 → 온보딩(3~5문항) → 메인(카드 스택 3개) → 스와이프/선택 → 로깅 완료 → 메인 복귀
```

### 5.2 재방문 사용자

```
앱 실행 → 메인(추천) 화면 → (이력 반영된 추천) → 선택 or 피드백 → 로깅
```

### 5.3 설정 변경

```
메인 → 설정 → 식사시간/제한사항/선호 수정 → 저장 → 메인 복귀
```

## 11. 부록

### 11.1 음식 DB 스키마 (초안)

- **저장:** CoreData 등 DB 사용. 하드코딩 금지.
- **이미지:** `imagePath` 또는 DB 내 이미지 저장. 운영자가 직접 등록.

```json
{
  "id": "string",
  "name": "string",
  "category": "korean|chinese|japanese|western|etc",
  "mealTypes": ["breakfast", "lunch", "dinner"],
  "restrictions": ["vegetarian", "glutenFree", "lowCalorie", "none"],
  "tags": ["string"],
  "baseScore": 1.0,
  "imagePath": null
}
```

### 11.2 프로파일 스키마 (초안)

```json
{
  "mealPattern": ["breakfast", "lunch", "dinner"],
  "restrictions": ["string"],
  "preferredCategories": ["string"],
  "customMealTime": {"start": "HH:mm", "end": "HH:mm"},
  "dislikes": ["string"]
}
```
