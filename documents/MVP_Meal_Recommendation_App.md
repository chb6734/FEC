# 식사 추천 앱 MVP 결과물

> **프로젝트:** 식사시간 맞춤형 음식 추천 iOS 애플리케이션  
> **토론 참여:** Elon CEO ↔ Athena PM  
> **작성일:** 2025-03-04  
> **목적:** 개인 패턴 학습 기반 식사 추천 앱 MVP 범위 및 방향성 확정

---

## 1. MVP 핵심 정의

### 1.1 제품 한 줄 정의
> **"식사시간에 맞춰, 사용자의 과거 패턴을 학습하여 개인화된 음식을 추천하는 iOS 앱"**

### 1.2 MVP 범위 (합의 결과)

| 구분 | 포함 | 제외 (Phase 2+) |
|------|------|-----------------|
| **핵심 기능** | 식사시간대별 추천, 수동 식사 로깅, 취향 반영 추천 | AI 기반 자동 학습, 카메라 음식 인식 |
| **데이터** | 로컬 저장 우선, 사용자 입력 기반 패턴 | 서버 동기화, 외부 API 연동 |
| **개인화** | 3가지 이상 취향 필터, 시간대별 선호도 | 깊은 ML 모델, 행동 예측 |
| **음식 DB** | DB 저장 (하드코딩 금지), 50~100개, 이미지 경로·추후 저장 | 대규모 DB, 실시간 메뉴 API |

## 2. 주요 결정사항

### 2.1 Cold Start 전략

**결정:** 온보딩 시 **3~5개 질문**으로 초기 프로파일 수집 후, 즉시 추천 제공 가능하게 구성

| 관점 | CEO | PM |
|------|-----|-----|
| **입장** | 첫 사용부터 추천 노출, "빈 화면 = 이탈" | 질문 5개 초과 시 드롭오프 급증 데이터 근거 |
| **합의** | 필수 3문항(식사 패턴, 제한사항, 선호 요리군) + 선택 2문항 |

### 2.2 추천 알고리즘

**결정:** MVP는 **규칙 기반(Rule-based)** 추천. ML은 Phase 2로 이관.

| 관점 | CEO | PM |
|------|-----|-----|
| **입장** | "학습"이라는 차별점 강조, ML 고려 | 데이터 0→1000건 축적 전 ML 의미 없음, 우선 로직 검증 |
| **합의** | 점수화 규칙(시간대 가중치 + 취향 매칭 + 최근 선택 회피)으로 구현. "학습"은 "선택 이력 축적 → 가중치 조정"으로 표현 |

### 2.3 식사시간 정의

**결정:** 고정 3타임(아침 6–10시, 점심 11–14시, 저녁 17–21시) + **사용자 커스텀 1개** 허용

| 관점 | CEO | PM |
|------|-----|-----|
| **입장** | 사용자 정의 시간대 유연성 | MVP 복잡도·테스트 부담 증가 |
| **합의** | 기본 3타임 고정, 설정에서 "나만의 식사시간" 1개 추가 가능 |

### 2.4 음식 데이터 소스

**결정:** MVP는 **DB 저장 필수** (CoreData 등). 하드코딩(JSON 번들, plist 등) 금지. 음식 데이터·이미지 모두 DB에 저장.

| 관점 | CEO | PM |
|------|-----|-----|
| **입장** | 외부 API로 품질·다양성 확보 | API 비용·의존성·지연, MVP 검증에 불필요 |
| **합의** | DB에 음식 데이터 저장. 이미지 경로·파일은 운영자가 직접 DB에 등록. 추후 API는 확장 옵션으로 검토 |

### 2.5 추천 UI (카드·스와이프)

**결정:** 추천을 **카드형 스택**으로 **정확히 3개** 노출. 틴더 스타일 스와이프 인터랙션.

| 항목 | 내용 |
|------|------|
| **추천 개수** | 3개 고정 |
| **카드 구성** | 중앙: 음식 사진/그래픽, 주변: 색상 테두리 |
| **인터랙션** | 좌/우 스와이프로 카드 넘기기 (틴더 스타일) |
| **이미지** | 운영자가 DB에 저장. MVP 시점 placeholder 대응 가능 |

## 6. MVP 기능 명세 (우선순위)

### P0 (필수)

1. **온보딩** – 3~5문항 프로파일 수집
2. **식사시간 추천** – 현재 시간대 기준 **카드형 3개** 추천, 틴더 스타일 스와이프 (중앙 이미지 + 색상 테두리)
3. **식사 로깅** – 추천 카드에서 선택 or 직접 입력
4. **로컬 DB** – 음식 50~100개 **DB 저장**(하드코딩 금지), 이미지 DB 저장, 규칙 기반 추천 엔진

### P1 (MVP 포함 권장)

5. **선호도 조정** – "맘에 들어요 / 별로예요" 피드백
6. **설정** – 식사시간 커스텀, 알레르기/제한사항 수정

### P2 (Phase 2)

7. 서버 동기화  
8. ML 기반 추천  
9. 푸시 알림  
10. 소셜/공유 기능
